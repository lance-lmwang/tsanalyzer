# TsAnalyzer: 引擎架构优化与技术增强计划

## 1. 背景与目标

在对业界优秀工具集和现有 `tsanalyzer` 引擎进行综合对比分析后，我们发现 `tsanalyzer` 定位于高性能、确定性的度量级分析引擎，可以通过吸收一些灵活的微工具架构和长效预测算法来进一步提升其产品力。

本文档总结了优秀的架构、算法及检测技术，并制定了将其引入到 `tsanalyzer` 现有产品体系中的分阶段实施计划，旨在进一步提升 TsAnalyzer 的可维护性、预测能力以及深层业务分析能力。

---

## 2. 核心增强点总结

### 2.1 架构与设计思想 (Architecture)
*   **高密度终端 TUI 界面**：引入使用 Ncurses 构建的直观终端流监控界面，在 NOC 无桌面环境下提供类似 `top` 命令的快速排障能力。
*   **分布式微工具链**：将 TsAnalyzer 底层核心库进一步抽象封装，提供更多法证切割与回放的离线微工具。

### 2.2 核心算法 (Algorithms)
*   **Jitter Buffer 抖动平滑**：通过 PCR 斜率推算与环形缓冲区重塑突发流量，为 TsAnalyzer 未来增加“主动修复”与“整形”能力打下基础。
*   **长期 PCR 时钟线性预测**：引入线性回归模型，强化 T-STD/VBV 缓冲区的长期溢出预测，实现提早数小时的预警。
*   **物理层 IAT 调度基准**：建立纯净的包到达间隔测量机制，将其引入系统探针自检脚本，量化评估内核调度与网卡亲和性环境。

### 2.3 检测与深度分析技术 (Detection Techniques)
*   **端到端闭环自测**：实现自生成参考流并在终端拦截校验的逻辑，增强 TsAnalyzer 自身的极限压力和准入验收测试。
*   **深层负荷与元数据分析**：扩展至 SCTE-35 广告标记、CEA-708 字幕抓取以及非注册 SEI 分析，从协议结构层 (PSI/SI) 向商业化内容感知层延伸。

### 2.4 Astra 高级架构借鉴 (Astra Architecture Integration)
*   **流处理树状架构 (Stream Tree Architecture)**：采用分层的、类似发布-订阅的树状流处理模型。数据源作为父节点，将 TS 包通过轻量级函数指针直接分发给多个子节点（独立的分析器模块）。该机制彻底解耦数据流，避免了冗余解析和内存拷贝。
*   **响应式 PID 过滤管理 (Reactive PID Management)**：支持按需订阅的 PID 机制，下游模块向上游注册所需 PID，底层的解复用阶段或数据摄入层在接收到未被请求的 PID 时直接丢弃，极大降低高码率 MPTS 流的 CPU 与内存总线消耗。
*   **高性能跨平台事件循环 (Reactor Pattern)**：采用单线程/少线程加上异步 I/O 多路复用的事件驱动模型，减少高并发情况下的线程上下文切换开销，保证主循环极低延迟。
*   **Lua 与 C 混合架构 (Lua-C Hybrid)**：引入轻量级脚本引擎（如 Lua）用于高层配置解析、业务逻辑编排与流媒体路由建立，底层性能敏感的数据解析依然保留 C 实现，极大增强系统架构的业务灵活性。
*   **插件化模块设计 (Module System)**：加强功能的模块化接口定义（Plugin API），利用注册机制将特定分析功能（如 SCTE-35 解析等）作为独立插件管理。

## 3. 分阶段实施计划

为了平稳且高效地落地这些技术增强，我们将其划分为以下四个阶段进行整合：

### 阶段一：短期计划 —— 完善基准与测试套件
**目标**：提升 TsAnalyzer 在部署前对环境和自身性能的量化评估能力。
*   **任务 1.1**：开发纯净的网络 IAT（包到达间隔）自测工具或脚本模块，集成到 `appliance_boot_verify.sh` 中。
*   **任务 1.2**：在 `tools/` 目录下开发一个高速参考 TS 流发生器 (`tsa_generator`)，用于在 1Gbps 线速下进行闭环自发自收验收。

### 阶段二：中期计划 —— 构建高密度终端可视化 (已完成)
**目标**：为本地运维提供实时的可视化诊断利器。
*   **任务 2.1**：开发 `tsa_top` 工具。实现基于 Ncurses 的类 `top` 交互界面，支持双行显示模式以平衡监控密度与诊断深度。
*   **任务 2.2**：建立高效 IPC 机制。采用 POSIX 共享内存实现 `tsa_server_pro` 与 `tsa_top` 之间的零拷贝指标传递。
*   **展示指标详情**：
    *   **基础指标**：Bitrate (Mbps), Health Score, CC Errors, TR 101 290 (P1/P2/P3) 统计。
    *   **确定性度量**：PCR Jitter (p99), MDI Delay Factor (ms)。
    *   **预测性诊断**：RST (Remaining Safe Time) 网络/编码器预警，PCR 时钟短期/长期漂移 (ppm)。
    *   **业务深探**：视频分辨率、帧率、GOP 长度、SCTE-35 广告标记检测、CEA-708 字幕检测。

### 阶段三：长期计划 —— 预测分析与深度业务感知 (已完成)
**目标**：实现 TsAnalyzer 从“被动监控”向“智能预测与主动修复”的跨越。
*   **任务 3.1（预测能力）**：吸收长期时钟漂移线性回归算法，增强 RST (Remaining Safe Time) 的长效预警能力。采用了相位差法 (Phase Difference) 与均值中心化回归，提供了极高的数值稳定性。
*   **任务 3.2（业务深探）**：开发对 SCTE-35、CEA-708 和 SEI 等深层净荷内容的专用解析模块。实现了详细的 SCTE-35 Splice Insert 指令解析，支持 EventID、持续时间和 PTS 时间戳提取。
*   **任务 3.3（主动修复落地）**：在 `tsa_gateway` 中实现了主动 PCR 重印 (Re-stamping) 和码率整形逻辑。在极端网络抖动环境下，能够将非标准源流重塑为完美的 CBR 广播流。
*   **配套工具链**：新增 `tsa_slicer` 法证切割工具，支持基于 PCR 时间轴的毫秒级离线切割。

### 阶段四：长期计划 —— Astra 架构融合与演进
**目标**：重构底层分发层，引入高并发轻量级架构以及动态可编程逻辑，支撑极大密度的流媒体监控探测。
*   **任务 4.1（重构流分发框架）**：引入 Stream Tree 观察者机制，使多个分析模块无冗余、零拷贝地共享底层摄入流的 TS 数据。
*   **任务 4.2（实现按需解析与过滤）**：引入 Reactive PID 机制，实现根据不同探测任务的需求向上游摄入引擎动态注册所需的过滤 PID。
*   **任务 4.3（引入事件循环）**：从固定多线程流水线探索切换至基于 Reactor 模式的高性能事件驱动循环以适应海量并发探测要求。
*   **任务 4.4（集成混合编程引擎）**：集成轻量级脚本语言（如 Lua 或 QuickJS）重塑控制面，将灵活多变的探测告警、配置生成规则提取至脚本层。
