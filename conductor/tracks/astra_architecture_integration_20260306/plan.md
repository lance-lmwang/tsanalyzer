# Implementation Plan: Dynamic Lua Pipeline (Y-Architecture)

This plan details the step-by-step approach to building the highly programmable Lua gateway/pipeline engine.

## Step 1: Core Lua Runtime Initialization & Context (DONE)
- **Goal**: Establish the basic Lua environment embedded inside the C engine.
- **Tasks**:
  - Update `include/tsa_lua.h` and `src/tsa_lua.c` to wrap `luaL_newstate()`, `luaL_openlibs()`.
  - Create a global `tsa` module table in Lua to hold all factory functions.
  - Implement the CLI entry point `tsanalyzer run <script.lua>`.

## Step 2: Stream Object Binding (Input & Output) (DONE)
- **Goal**: Expose basic network I/O to Lua.
- **Tasks**:
  - Create Lua userdata bindings for `tsa_source_t` (UDP/SRT input) and `tsp_handle_t` (UDP/SRT output).
  - Define metatables with `__gc` for safe destruction.

## Step 3: The Analyzer Binding & Upstream Topological Linking (Current Focus)
- **Goal**: Expose the core `tsa_handle_t` (Analyzer) to Lua and implement the graph linking mechanism so data can flow seamlessly from Input -> Analyzer -> Output.
- **Implementation Steps**:
  1.  **C-Side Stream API**: Ensure `tsa_stream.c` has robust `tsa_stream_add_child()` and `tsa_stream_remove_child()` functions.
  2.  **Lua Analyzer Factory**: Implement `tsa.analyzer()` in `tsa_lua.c` which instantiates a `tsa_handle_t` and wraps it in a Lua Userdata (`tsa.analyzer` metatable).
  3.  **Metatable Method (`set_upstream`)**: Implement `analyzer:set_upstream(source)` and `output:set_upstream(analyzer)`.
      - When called, the C code extracts the raw C pointers from the Lua Userdata arguments and links their stream hooks (`tsa_stream_add_child`).
  4.  **Graceful Teardown**: Update all `__gc` meta-methods to automatically detach from their parents/children before destroying the C objects to prevent dangling pointers.
- **Test Method**:
  - Write `tests/test_lua_pipeline.c` which invokes a script creating a UDP input, an analyzer, and a UDP output, links them, feeds a few mock TS packets into the input, and verifies the analyzer's packet counters increase.

## Step 4: Reactive PID Filtering & Routing
- **Goal**: Allow Lua scripts to dynamically subscribe to specific PIDs, dropping unwanted traffic at the earliest possible stage to save CPU.
- **Implementation Steps**:
  1.  **Lua Method**: Implement `analyzer:join_pid(pid)` and `analyzer:drop_pid(pid)`.
  2.  **C-Side Engine**: Hook these into the `h->pid_filters` or `h->pid_seen` logic within `tsa_handle_t`. If a Lua script joins a PID, the analyzer begins processing it; otherwise, it bypasses full metrology.
- **Test Method**:
  - Modify `test_lua_pipeline.c` to instantiate an analyzer, call `join_pid(0x100)` via script, feed packets for 0x100 and 0x200, and assert that only stats for 0x100 are updated.

## Step 5: Event Callbacks (Alarms to Lua)
- **Goal**: Allow Lua to react asynchronously to alarms generated by the C metrology engine (e.g., triggering failover logic).
- **Implementation Steps**:
  1.  **Lua Event Registry**: Implement `analyzer:on("event_name", callback_func)`.
  2.  **C-to-Lua Bridge**: In `tsa_alert.c` or `tsa.c` (where `tsa_push_event` resides), if a Lua callback is registered for that event type, push the event data (PID, error type, value) onto the Lua stack and invoke the function safely (`lua_pcall`).
- **Test Method**:
  - Create a Lua script that registers `analyzer:on("SYNC_LOSS", handler)`. In the C test framework, intentionally stop feeding packets or feed garbage. Assert that the Lua handler was successfully executed (e.g., by having the Lua script flip a global flag accessible via C API).

## Release Criteria
- **Full E2E Validation**: The test suite must pass `make full-test` with zero memory leaks (`valgrind` clean) across Lua garbage collection cycles.
- **Zero-Allocation Data Plane**: Ensure that `set_upstream` linking uses standard function pointers (no `malloc` per packet during routing).
